4. REGLAS TÉCNICAS IMPLÍCITAS DEL MODELO

Regla: Inmutabilidad por ciclo
Descripción:
Una vez generado el snapshot de Integration para un ciclo, no se modifica
Impacto:
Permite reproducibilidad total

---

Regla: Separación Integration vs Serving
Descripción:
Integration representa datos base
Serving representa interpretación del modelo
Impacto:
Permite múltiples corridas sobre misma base

---

Regla: Resolución de dominio
Descripción:
Un dominio puede tener múltiples advertisers
Se resuelve usando:

1. prioridad de product_status
2. fecha de contratación
3. si empate → excepción
   Impacto:
   Evita duplicación de métricas

---

Regla: Exclusión por excepción de dominio
Descripción:
Si dominio no se puede resolver → no se usa en métricas
Impacto:
Protege cálculos de FL

---

Regla: Riesgo indeterminado
Descripción:
Es una etiqueta formal del modelo
Sustituye cualquier otra clasificación
Impacto:
Simplifica consumo en API/UI

---

Regla: Ventana de cases
Descripción:
Se persisten 120 días
El motor decide qué usar
Impacto:
Flexibilidad analítica

---

Regla: Producto relevante
Descripción:
Solo se consideran estatus L, I, D, S
Impacto:
Reduce volumen y ruido

---

Regla: Dominio como llave de trazabilidad
Descripción:
Se usa domain_normalized
No bc_product_id
Impacto:
Soporta cambios de negocio

---

Regla: Parsing de advertiser-business
Descripción:
Campo compuesto se separa en:

* advertiser_id
* business_id
  Impacto:
  Evita dependencia de formato fuente

5. PATRONES DE EJECUCIÓN

Patrón: Carga por fuente
Descripción:
Cada fuente genera un source_load_id independiente

---

Patrón: Batch maestro
Descripción:
Agrupa múltiples cargas

---

Patrón: Snapshot Integration
Descripción:
Se genera una vez por ciclo

---

Patrón: Corrida del motor
Descripción:
Lee Integration + Parameter Version
Genera Serving

---

Patrón: Publicación
Descripción:
Una corrida por ciclo se marca como vigente

---

Patrón: Consumo
Descripción:
API/UI consume vistas filtradas por is_published

---

6. CONSIDERACIONES DE PERFORMANCE

* Uso intensivo de índices por cycle_id
* Evitar joins a DB links en runtime del motor
* Integration actúa como cache persistente
* Parámetros deben cachearse en PL/SQL
* Evitar conversiones de tipos NUMBER/VARCHAR
* Tablas de detalle diseñadas para inserts masivos
